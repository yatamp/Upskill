AI, LLMs, Agents & MCP — Deep Dive

Level: Beginner → Advanced  ·  How it all works under the hood + how to build real things

How LLMs Actually Work — From Scratch

A Large Language Model is a probability distribution over tokens. Given a sequence of tokens, it predicts the probability of each possible next token. That's the core — everything else is scale and technique.


Tokenisation

Text is split into tokens (sub-word units) using BPE (Byte-Pair Encoding). Common words are single tokens; rare words split into sub-tokens. "unbelievable" → ["un", "believ", "able"]. Vocabulary size ≈ 100K tokens. Token ≈ 0.75 words on average.

Embeddings

Each token is mapped to a high-dimensional vector (e.g. 4096 dimensions). Semantically similar words end up close in this space — "king" - "man" + "woman" ≈ "queen". These are learned during training.

Transformer Layers

Stack of identical layers. Each layer has: Multi-Head Self-Attention (each token attends to all others, learning relationships) + Feed-Forward Network (per-token transformation). Deeper = higher-level abstractions.

Self-Attention

For each token, compute attention scores against all other tokens (Query × Key matrix). Weight the Value vectors by those scores. Result: each token's representation is updated with context from the whole sequence. This is how "it" knows "it" refers to "the cat".

Context Window

Maximum number of tokens the model can process at once. Everything outside the window is invisible. Models have no memory — each API call is stateless. You must include all relevant context in the prompt.

Training

Pre-training: predict next token on trillions of tokens of text — learns world knowledge and language structure. Fine-tuning + RLHF (Reinforcement Learning from Human Feedback): specialise for instruction following and helpfulness. Constitutional AI (Anthropic): align using a set of principles.

Inference

Generating output token by token. Each token is sampled from the probability distribution. Temperature controls randomness. Top-p (nucleus sampling) limits to highest-probability tokens. Greedy = always pick top-1 = deterministic but can loop.

Emergent Capabilities

Abilities not explicitly trained that appear above certain model scales: few-shot learning, chain-of-thought reasoning, code generation, arithmetic. This "emergence" is why larger models aren't just incrementally better — they're qualitatively different.

Prompt Engineering — Getting Reliable Output

Be explicit about format

LLMs follow format instructions well. "Respond in JSON with keys: status, message, code" → reliable structured output. Use XML tags to delimit sections: <context>...</context>.

Chain-of-Thought (CoT)

"Think step by step before answering." Forces the model to reason through the problem. Dramatically improves accuracy on multi-step problems. Extended thinking (Claude) does this internally.

Few-shot examples

Show 2–3 examples of input→output before your actual input. The model infers the pattern. Works especially well for formatting, tone, classification tasks.

Role + context

"You are a senior DevOps engineer reviewing Terraform code for security issues." Sets the lens. More specific = better. Include relevant constraints: "production-grade", "explain to a junior", "be concise".

Grounding

Provide the actual data in the prompt rather than relying on model knowledge. "Here is the error log: [PASTE]. Diagnose the root cause." Eliminates hallucination about specifics.

Temperature control

0 = deterministic (code, facts, structured output). 0.3–0.7 = balanced (analysis, summaries). 0.8–1.0 = creative (brainstorming, writing). Always use 0 in production pipelines.


Prompt template for reliable structured output:

SYSTEM:
You are a DevOps expert. Analyse the provided CI logs and respond ONLY with valid JSON.
No explanation, no markdown, just JSON.

USER:

{paste logs here}


Respond with:
{
  "root_cause": "one sentence explanation",
  "severity": "low|medium|high|critical",
  "failing_step": "step name",
  "suggested_fix": "specific actionable fix",
  "confidence": 0.0-1.0
}

RAG — Retrieval Augmented Generation

LLMs have a knowledge cutoff and can't access your private data. RAG solves this: retrieve relevant documents at query time, inject them into the prompt.


User query: "How do I rotate the DB credentials in prod?"
   ↓
Embed query → vector [0.23, -0.41, 0.87, ...]
   ↓
Vector similarity search in your doc database
(runbooks, wikis, Confluence, Slack, code)
   ↓
Retrieve top-5 most relevant chunks
   ↓
Inject into prompt: "Using this context: [chunks], answer: [query]"
   ↓
LLM generates answer grounded in YOUR documentation

Embeddings

Turn text into vectors. Semantically similar text → similar vectors. Use: text-embedding-3-small (OpenAI), Cohere Embed v3, or local models (nomic-embed-text). Embed everything once, store in vector DB.

Vector Databases

Stores embeddings, performs fast similarity search. Options: Pinecone, Weaviate, Qdrant (open-source), pgvector (Postgres extension), Chroma (local dev). Choose based on scale and existing infra.

Chunking strategy

Split documents into chunks before embedding. Too small = no context. Too large = noise. Good defaults: 512 tokens with 50-token overlap. Preserve semantic units — don't split mid-sentence or mid-code-block.

Re-ranking

After initial retrieval, run a cross-encoder model to re-rank by true relevance. Cohere Rerank, Jina Reranker. Dramatically improves quality. Retrieve top-20, re-rank to top-5 before sending to LLM.


Minimal RAG with pgvector + Python:

import anthropic
import psycopg2
import numpy as np

client = anthropic.Anthropic()

def get_embedding(text: str) -> list[float]:
    """Get embedding using Claude's embedding endpoint."""
    # Or use OpenAI/Cohere embeddings
    import openai
    response = openai.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def retrieve_context(query: str, conn, top_k: int = 5) -> list[str]:
    """Find most relevant chunks from your knowledge base."""
    query_embedding = get_embedding(query)
    cursor = conn.cursor()
    cursor.execute("""
        SELECT content, 1 - (embedding <=> %s::vector) AS similarity
        FROM documents
        ORDER BY similarity DESC
        LIMIT %s
    """, (query_embedding, top_k))
    return [row[0] for row in cursor.fetchall()]

def rag_query(question: str, conn) -> str:
    """Answer a question using retrieved context."""
    chunks = retrieve_context(question, conn)
    context = "\n\n---\n\n".join(chunks)

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="""You are a helpful assistant. Answer questions using ONLY
        the provided context. If the answer isn't in the context, say so.""",
        messages=[{
            "role": "user",
            "content": f"\n{context}\n\n\nQuestion: {question}"
        }]
    )
    return response.content[0].text

Agent Patterns — How to Build Reliable AI Agents

From Anthropic's research — build the simplest architecture that works. Add complexity only when needed.


Pattern 1: Prompt Chaining

Break a complex task into sequential steps. Output of each step is input to the next. Each step uses a focused prompt. Good for: multi-stage pipelines where each step can be verified.

When: tasks with clear sequential stages — generate → review → refine → format
# Step 1: Extract key info
analysis = claude("Analyse this PR diff and list potential issues: " + diff)

# Step 2: Research each issue
for issue in parse_issues(analysis):
    research = claude("Explain why this pattern is problematic: " + issue)

# Step 3: Generate recommendations
recommendations = claude("Based on these issues and research, write review comments: "
                         + analysis + research)

Pattern 2: Routing

A classifier model decides which specialised sub-agent or prompt handles the input. Keeps each agent focused and better at its specific task.

When: input can be categorised — different handling for billing vs tech support vs escalations
def route_request(user_message: str) -> str:
    route = claude(f"""Classify this request into one of:
    billing, technical, account, other.
    Reply with ONLY the category.
    Request: {user_message}""")

    handlers = {
        "billing": billing_agent,
        "technical": technical_agent,
        "account": account_agent,
    }
    return handlers.get(route.strip(), general_agent)(user_message)

Pattern 3: Parallelisation

Run multiple independent LLM calls in parallel, then aggregate results. Dramatically faster than sequential. Two sub-patterns: sectioning (split task into parallel sub-tasks) and voting (multiple independent answers, take majority).

When: independent sub-tasks, or when you want higher confidence via voting
import asyncio

async def analyse_pr_parallel(diff: str) -> dict:
    """Run security, performance, and style checks in parallel."""
    security_task = asyncio.create_task(
        async_claude("Review for security issues: " + diff)
    )
    perf_task = asyncio.create_task(
        async_claude("Review for performance issues: " + diff)
    )
    style_task = asyncio.create_task(
        async_claude("Review for code style issues: " + diff)
    )
    security, perf, style = await asyncio.gather(security_task, perf_task, style_task)
    return {"security": security, "performance": perf, "style": style}

Pattern 4: Orchestrator + Sub-agents

One orchestrator LLM plans and delegates; specialised sub-agents execute. The orchestrator synthesises results. This is how Claude Code works — an orchestrator spawning Explore, Plan, and coding sub-agents.

When: complex tasks requiring planning + specialised execution across domains
def orchestrator_agent(task: str) -> str:
    # Orchestrator plans the work
    plan = claude(f"""Break this task into subtasks for specialist agents:
    Available agents: [code_reviewer, security_scanner, doc_writer, test_generator]
    Task: {task}
    Return a JSON list of {{agent, input}} pairs.""")

    subtasks = json.loads(plan)
    results = {}

    for subtask in subtasks:
        agent_fn = AGENTS[subtask["agent"]]
        results[subtask["agent"]] = agent_fn(subtask["input"])

    # Orchestrator synthesises
    return claude(f"Synthesise these results into a final report: {results}")

Pattern 5: Evaluator-Optimizer Loop

One LLM generates output; another evaluates it and provides feedback; generator refines. Iterate until quality threshold met. Produces higher quality output than single-shot generation.

When: quality matters more than speed — writing, code generation, data extraction
def generate_with_feedback(task: str, max_rounds: int = 3) -> str:
    output = claude("Generate: " + task)

    for round in range(max_rounds):
        evaluation = claude(f"""Evaluate this output for the task: {task}
        Output: {output}
        Respond with JSON: {{"score": 1-10, "issues": [], "pass": true/false}}""")

        eval_data = json.loads(evaluation)
        if eval_data["pass"] or eval_data["score"] >= 8:
            break

        output = claude(f"""Improve this output addressing these issues:
        Issues: {eval_data['issues']}
        Current output: {output}""")

    return output

MCP — Model Context Protocol Deep Dive

MCP is an open standard for connecting AI models to external systems. It separates what the model can do from how the model reasons.


┌─────────────────────────────────────┐
│ MCP HOST │
│ (Claude Desktop / Claude Code / │
│ your app using MCP client SDK) │
└─────────────┬───────────────────────┘
             │ JSON-RPC 2.0
  stdio transport (local) or HTTP+SSE
             │
┌─────────────▼───────────────────────┐
│ MCP SERVER │
│ Exposes: Tools / Resources / │
│ Prompts / Sampling │
└─────────────┬───────────────────────┘
             │
┌─────────────▼───────────────────────┐
│ External System │
│ (filesystem, DB, GitHub, │
│ Kubernetes, AWS, Slack...) │
└─────────────────────────────────────┘

MCP primitives explained:

PrimitiveDirectionDescriptionExample

ResourcesModel → ServerRead-only data the model can access. Application-controlled what to expose.file:///src/, db://users/schema

Build a production MCP server — DevOps tools (Python):

#!/usr/bin/env python3
"""
MCP Server: DevOps Tools
Exposes: Kubernetes, deployment status, log access
"""
import json, subprocess, os
from datetime import datetime
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("DevOps Tools", version="1.0.0")

# ── TOOLS ──────────────────────────────────────────────────

@mcp.tool()
def kubectl_get(resource: str, namespace: str = "default") -> str:
    """
    Get Kubernetes resources.
    Args:
        resource: Resource type and optional name (e.g. 'pods', 'pods/mypod', 'deployments')
        namespace: Kubernetes namespace (default: 'default')
    """
    result = subprocess.run(
        ["kubectl", "get", resource, "-n", namespace, "-o", "json"],
        capture_output=True, text=True, timeout=30
    )
    if result.returncode != 0:
        return f"Error: {result.stderr}"
    data = json.loads(result.stdout)
    # Simplify output
    if "items" in data:
        return json.dumps([{
            "name": item["metadata"]["name"],
            "status": item.get("status", {}).get("phase", "unknown"),
            "ready": item.get("status", {}).get("containerStatuses", [{}])[0].get("ready", False)
            if item.get("kind") == "Pod" else None
        } for item in data["items"]], indent=2)
    return result.stdout

@mcp.tool()
def get_pod_logs(pod_name: str, namespace: str = "default",
                 lines: int = 100, container: str = None) -> str:
    """
    Get logs from a Kubernetes pod.
    Args:
        pod_name: Name of the pod
        namespace: Kubernetes namespace
        lines: Number of tail lines (default: 100)
        container: Specific container name (for multi-container pods)
    """
    cmd = ["kubectl", "logs", pod_name, "-n", namespace, f"--tail={lines}"]
    if container:
        cmd.extend(["-c", container])
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=30)
    return result.stdout if result.returncode == 0 else f"Error: {result.stderr}"

@mcp.tool()
def rollout_status(deployment: str, namespace: str = "default") -> str:
    """Check the rollout status of a deployment."""
    result = subprocess.run(
        ["kubectl", "rollout", "status", f"deployment/{deployment}",
         "-n", namespace, "--timeout=5s"],
        capture_output=True, text=True
    )
    return result.stdout or result.stderr

@mcp.tool()
def rollback_deployment(deployment: str, namespace: str = "default") -> str:
    """
    Roll back a deployment to the previous version.
    CAUTION: This is a destructive operation.
    Args:
        deployment: Name of the deployment to roll back
        namespace: Kubernetes namespace
    """
    result = subprocess.run(
        ["kubectl", "rollout", "undo", f"deployment/{deployment}", "-n", namespace],
        capture_output=True, text=True
    )
    return result.stdout if result.returncode == 0 else f"Error: {result.stderr}"

@mcp.tool()
def get_cluster_events(namespace: str = "default", limit: int = 20) -> str:
    """Get recent Kubernetes events, sorted by time."""
    result = subprocess.run(
        ["kubectl", "get", "events", "-n", namespace,
         "--sort-by=.lastTimestamp", "-o", "json"],
        capture_output=True, text=True, timeout=30
    )
    if result.returncode != 0:
        return f"Error: {result.stderr}"
    data = json.loads(result.stdout)
    events = data.get("items", [])[-limit:]
    return json.dumps([{
        "time": e["lastTimestamp"],
        "type": e["type"],
        "reason": e["reason"],
        "message": e["message"],
        "object": f"{e['involvedObject']['kind']}/{e['involvedObject']['name']}"
    } for e in events], indent=2)

# ── RESOURCES ─────────────────────────────────────────────

@mcp.resource("cluster://namespaces")
def list_namespaces() -> str:
    """List all Kubernetes namespaces."""
    result = subprocess.run(
        ["kubectl", "get", "namespaces", "-o", "jsonpath={.items[*].metadata.name}"],
        capture_output=True, text=True
    )
    return result.stdout

@mcp.resource("logs://system/{service}")
def get_system_logs(service: str) -> str:
    """Get systemd service logs."""
    result = subprocess.run(
        ["journalctl", "-u", service, "--no-pager", "-n", "50"],
        capture_output=True, text=True
    )
    return result.stdout

# ── PROMPTS ────────────────────────────────────────────────

@mcp.prompt()
def incident_response(service: str, symptom: str) -> str:
    """Generate an incident response investigation prompt."""
    return f"""You are an SRE investigating an incident.

Service: {service}
Reported symptom: {symptom}
Time: {datetime.now().isoformat()}

Investigation steps:
1. Use kubectl_get to check pod status for {service}
2. Use get_pod_logs to check for errors in the last 200 lines
3. Use get_cluster_events to see recent Kubernetes events
4. Use rollout_status to check if a deployment is in progress

Provide:
- Root cause assessment
- Immediate mitigation steps
- Whether a rollback is warranted"""

if __name__ == "__main__":
    mcp.run()

Extended Thinking — Claude's Internal Reasoning

Claude's extended thinking mode lets the model reason through a problem before producing its final answer. The thinking is visible to you (as a thinking block) but doesn't count toward output costs.

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-6",          # Extended thinking on Opus 4.6
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000        # How many tokens to use for thinking
    },
    messages=[{
        "role": "user",
        "content": """Design a database schema for a multi-tenant SaaS application
        with teams, users, projects, billing, and audit logs. Consider:
        - Row-level security
        - Performance at 10M users
        - Tenant isolation approaches
        Return the full schema with rationale for each decision."""
    }]
)

for block in response.content:
    if block.type == "thinking":
        print("=== Claude's reasoning ===")
        print(block.thinking[:500] + "...")   # Often very long
    elif block.type == "text":
        print("=== Final answer ===")
        print(block.text)
When to use extended thinking: Complex architecture decisions, multi-constraint optimisation, tasks requiring logical deduction across many factors. Not needed for simple tasks — adds latency and cost. Budget 5K–20K tokens of thinking for hard problems.

"""AI DevOps Assistant — uses Claude with tool use to answer infra questions."""
import anthropic, json, subprocess, os, sys
from rich.console import Console
from rich.markdown import Markdown

client = anthropic.Anthropic()
console = Console()

TOOLS = [
    {
        "name": "run_command",
        "description": "Run a safe, read-only shell command to gather system information. Only diagnostic commands allowed.",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string", "description": "The shell command to run"},
                "reason": {"type": "string", "description": "Why you need to run this command"}
            },
            "required": ["command", "reason"]
        }
    },
    {
        "name": "read_file",
        "description": "Read a file's contents",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "Absolute path to file"}
            },
            "required": ["path"]
        }
    }
]

ALLOWED_COMMANDS = ["docker", "kubectl", "ps", "df", "free", "uptime",
                    "ss", "netstat", "ping", "dig", "curl", "cat /var/log",
                    "systemctl status", "journalctl", "ls", "find", "echo"]

def is_safe_command(cmd: str) -> bool:
    return any(cmd.strip().startswith(allowed) for allowed in ALLOWED_COMMANDS)

def execute_tool(name: str, inputs: dict) -> str:
    if name == "run_command":
        cmd = inputs["command"]
        if not is_safe_command(cmd):
            return f"BLOCKED: '{cmd}' is not in the allowed command list."
        console.print(f"[dim]  Running: {cmd}[/dim]")
        result = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=15)
        return result.stdout or result.stderr or "(no output)"
    elif name == "read_file":
        path = inputs["path"]
        try:
            with open(path) as f:
                return f.read()[:5000]
        except Exception as e:
            return f"Error: {e}"
    return "Unknown tool"

def ask(question: str):
    console.print(f"\n[bold cyan]Question:[/bold cyan] {question}\n")
    messages = [{"role": "user", "content": question}]

    while True:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=4096,
            system="""You are a DevOps expert assistant. When asked about system state,
            use the available tools to gather real data rather than guessing.
            Be concise and actionable. If something looks wrong, say so clearly.""",
            tools=TOOLS,
            messages=messages
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, 'text'):
                    console.print(Markdown(block.text))
            break

        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    console.print(f"[dim yellow]  Using tool: {block.name}({list(block.input.keys())})[/dim yellow]")
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })
            messages.append({"role": "user", "content": tool_results})

if __name__ == "__main__":
    question = " ".join(sys.argv[1:]) if len(sys.argv) > 1 else input("Ask: ")
    ask(question)

RAG retrieves relevant documents at query time and injects them into the prompt. The model's weights don't change. Good for: dynamic data that changes frequently, private/internal knowledge, traceable sources. Fast to set up.
Fine-tuning trains the model on new data, updating its weights. Good for: changing the model's style/format/behaviour, teaching new patterns that appear constantly, making a smaller model match a larger one's quality for a specific task. Expensive and slow to iterate.

Q2. Your agent is getting stuck in a tool-use loop. How do you diagnose and fix it?

Show answer

(1) Log every tool call with inputs and outputs — find where the loop starts. (2) Add a max_iterations limit with a clear error message. (3) Check if the tool is returning useful information — if a tool fails silently, the model may keep retrying. (4) Check the system prompt — does it have a clear stopping condition? (5) Check if tool outputs are too long (truncate large outputs). (6) Consider if the model misunderstands the task — improve the system prompt to clarify when to stop.

Q3. What are the three MCP primitives and what's the key difference between Tools and Resources?

Show answer

Tools, Resources, and Prompts. Tools are functions the model calls to perform actions or get dynamic data — they can have side effects (write, execute, send). Model-controlled. Resources are read-only data sources exposed as URIs — no side effects. Application-controlled (the host decides what to surface). Use resources for static/slow-changing data the model should be aware of; use tools for actions and dynamic queries.

Q4. Why does temperature matter and what should you use for production pipelines?

Show answer

Temperature controls randomness in token sampling. Low temperature → model picks high-probability tokens → deterministic, consistent output. High temperature → more exploration of lower-probability tokens → creative but inconsistent. Production pipelines: always use 0. You want the same input to produce the same output reliably. Inconsistency in structured output (JSON, code) causes downstream failures. Reserve higher temperatures (0.7+) for creative tasks where variety is desirable and humans review the output.
