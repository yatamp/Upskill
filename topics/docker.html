Docker

Level: Beginner → Intermediate  ·  Estimated: 3–4 days  ·  The foundation of modern deployment

How Docker Works — Internals

Docker is not a VM. It uses Linux kernel features to isolate processes — containers share the host kernel.


Namespaces

Kernel feature that isolates: PID (process tree), NET (network stack), MNT (filesystem mounts), UTS (hostname), IPC (inter-process communication), USER (UID mapping). Each container gets its own namespace set.

cgroups

Control Groups — limit and account for resource usage. Docker uses cgroups to enforce CPU, memory, disk I/O limits per container. --memory 512m --cpus 0.5 sets cgroup limits.

Union Filesystem

Images are stacked read-only layers. Container adds a writable layer on top. Layers are cached and shared between images. overlay2 is the default storage driver on modern Linux.

Container vs VM

VM: full OS + kernel per instance, ~minutes to start, GB of disk. Container: shared kernel, process isolation, ~milliseconds to start, MB of disk. Containers are processes, not machines.

Docker Engine

dockerd (daemon) manages images/containers. Docker CLI talks to dockerd via REST API over a Unix socket (/var/run/docker.sock). containerd (lower-level runtime) does the actual container lifecycle. runc creates containers.

OCI Standard

Open Container Initiative defines the image format and runtime spec. Docker images are OCI-compliant — they run on containerd, Podman, Kubernetes (via CRI) without Docker itself.


Image layers (read-only):
┌─────────────────────────────────┐ ← Layer 4: COPY app/ /app/ (your code)
├─────────────────────────────────┤ ← Layer 3: RUN npm ci

├─────────────────────────────────┤ ← Layer 1: FROM node:20-alpine

Writing Good Dockerfiles

Production Node.js Dockerfile — best practices applied:

# syntax=docker/dockerfile:1
FROM node:20-alpine AS base
# Use Alpine for small images (~5MB vs ~900MB for node:20)
# Pin to major version, not 'latest'

WORKDIR /app

# ── deps stage ─────────────────────────────────
FROM base AS deps
# Copy ONLY package files first — separate layer
# This layer is cached unless package.json changes
COPY package.json package-lock.json ./
RUN npm ci --only=production

# ── build stage ────────────────────────────────
FROM base AS build
COPY package.json package-lock.json ./
RUN npm ci                          # Install dev deps too for building
COPY . .
RUN npm run build                   # Compile TypeScript, bundle, etc.

# ── final stage ────────────────────────────────
FROM base AS final

# Security: run as non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Copy only what's needed in production
COPY --from=deps /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
COPY package.json .

# Switch to non-root
USER appuser

# Document the port (doesn't actually expose it — that's -p flag)
EXPOSE 3000

# Health check — Docker/K8s can use this
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
    CMD wget -qO- http://localhost:3000/health || exit 1

# Use exec form (not shell form) — receives signals correctly
CMD ["node", "dist/server.js"]

Common Dockerfile instructions:

InstructionPurpose
FROMBase image. Always the first instruction. Use specific tags, never latest in production.
WORKDIRSet working directory for subsequent instructions. Creates it if absent.
COPYCopy files from build context into image. Prefer over ADD (ADD has magic tar extraction behaviour).
RUNExecute command and commit result as a new layer. Chain with && to keep in one layer.
ENVSet environment variable — available at build time AND runtime.
ARGBuild-time variable only. Pass with --build-arg. Not available in running container.
EXPOSEDocumentation — marks port the container listens on. Doesn't actually publish it.
USERSwitch to this user for subsequent RUN/CMD/ENTRYPOINT instructions.
ENTRYPOINTThe executable to run. CMD provides default arguments. Together: ENTRYPOINT ["npm"] CMD ["start"].
CMDDefault command. Overridden by docker run image YOUR_COMMAND.
HEALTHCHECKCommand Docker runs to check if container is healthy.
LABELMetadata key-value pairs. Good for: version, maintainer, build date.

Layer caching rules: Each instruction is a layer. If a layer changes, ALL subsequent layers are invalidated. Always put things that change least often first (OS packages → deps → source code). Never put secrets in a Dockerfile — they end up in image history even if deleted in a later layer.

.dockerignore — Keep Images Lean

# .dockerignore

.DS_Store
coverage/
.nyc_output/
Dockerfile*

Rule: If it's in .gitignore, it's probably in .dockerignore too. Small build contexts = faster builds.

Docker Compose — Multi-Container Apps

Compose defines multi-container apps in a single YAML file. Essential for local development.

# docker-compose.yml
version: '3.9'

services:
  app:
    build:
      context: .
      target: final            # Use specific stage in multi-stage build
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgres://postgres:secret@db:5432/myapp
      - REDIS_URL=redis://cache:6379
    env_file:
      - .env.local             # Load additional env vars from file
    depends_on:
      db:
        condition: service_healthy   # Wait for DB health check
      cache:
        condition: service_started
    volumes:
      - .:/app                 # Mount source for hot reload in dev
      - /app/node_modules      # Anonymous volume — don't overwrite container's node_modules
    restart: unless-stopped

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: secret
    volumes:
      - postgres_data:/var/lib/postgresql/data   # Named volume = persistent data
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql   # Init script
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5

  cache:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

  nginx:
    image: nginx:1.25-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/certs:/etc/nginx/certs:ro
    depends_on:
      - app

volumes:
  postgres_data:               # Named volumes persist across compose down/up

Container Security Best Practices

Never run as root

Use USER in Dockerfile. If breached, attacker is constrained to the container user. Map to non-root with --user 1000:1000 at runtime.

Read-only filesystem

docker run --read-only --tmpfs /tmp app. App can't write anywhere except explicitly mounted paths. Limits blast radius of a breach.

Drop capabilities

--cap-drop ALL --cap-add NET_BIND_SERVICE — drop all Linux capabilities, add back only what you need. Limits what a compromised container can do.

Scan images

docker scout cves myimage or trivy image myimage — scan for known CVEs in base image and dependencies. Add to CI pipeline.

Don't mount docker.sock

Mounting /var/run/docker.sock into a container gives it root on the host. Never do this in production.

Secrets management

Don't put secrets in images or environment variables if possible. Use Docker secrets (Swarm), K8s Secrets, or vault integrations. At minimum, use --env-file not hardcoded values.

Hands-On Project: Containerise a Full Stack App

1
Create the app 
mkdir docker-lab && cd docker-lab

# Simple Python Flask API
cat > app.py << 'EOF'
from flask import Flask, jsonify
import os, redis

app = Flask(__name__)
cache = redis.Redis(host=os.getenv('REDIS_HOST', 'localhost'), port=6379)

@app.route('/health')
def health():
    return jsonify(status='ok')

@app.route('/count')
def count():
    n = cache.incr('visits')
    return jsonify(visits=int(n))

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
EOF

cat > requirements.txt << 'EOF'
flask==3.0.0
redis==5.0.1
gunicorn==21.2.0
EOF
2
Write a production Dockerfile 
cat > Dockerfile << 'EOF'
FROM python:3.12-slim AS base
WORKDIR /app

FROM base AS deps
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

FROM base AS final
COPY --from=deps /usr/local/lib/python3.12/site-packages /usr/local/lib/python3.12/site-packages
COPY --from=deps /usr/local/bin/gunicorn /usr/local/bin/gunicorn
RUN adduser --disabled-password appuser
COPY --chown=appuser:appuser app.py .
USER appuser
EXPOSE 5000
HEALTHCHECK --interval=30s --timeout=3s CMD wget -qO- http://localhost:5000/health || exit 1
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "2", "app:app"]
EOF
3
Write docker-compose.yml 
cat > docker-compose.yml << 'EOF'
version: '3.9'
services:
  app:
    build: .
    ports: ["5000:5000"]
    environment:
      - REDIS_HOST=cache
    depends_on:
      - cache
  cache:
    image: redis:7-alpine
EOF
4
Build, run, test, inspect 
docker compose up -d --build

# Test the app
curl http://localhost:5000/health
curl http://localhost:5000/count   # increment counter
curl http://localhost:5000/count   # watch it go up

# Inspect the image layers
docker image history docker-lab-app

# Check resource usage
docker stats --no-stream

# Shell in
docker compose exec app sh
  > ps aux       # notice: gunicorn as non-root
  > id           # appuser
  > exit

# View logs
docker compose logs app

# Tear down
docker compose down
5
Test the layer cache in action 
# First build: all layers built fresh
docker build -t docker-lab-app .

# Change only app.py (not requirements.txt)
echo "# comment" >> app.py

# Second build: watch how requirements layer is CACHED
docker build -t docker-lab-app .
# Output: "CACHED" on the pip install layer — fast!

Check Your Understanding

Q1. Your Docker image is 1.2GB. What's your approach to reduce it?

Show answer

1. Use docker image history and dive to find large layers. 2. Switch to Alpine or slim base image. 3. Use multi-stage build — only copy final artifacts, not build tools. 4. Combine RUN commands with && and clean up in the same layer (rm -rf /var/lib/apt/lists/*). 5. Add a comprehensive .dockerignore. A Node.js app should be under 150MB, Python under 200MB.

Q2. What's the difference between ENTRYPOINT and CMD?

Show answer

ENTRYPOINT sets the executable — it's not easily overridden. CMD sets default arguments to ENTRYPOINT, OR the default command if no ENTRYPOINT. Common pattern: ENTRYPOINT ["python", "app.py"] — docker run myimage --debug passes --debug to python app.py. If you use only CMD, users can override the whole command: docker run myimage bash. Exec form (["cmd", "arg"]) vs shell form (cmd arg) matters for signal handling — always use exec form.

Q3. Why is mounting /var/run/docker.sock into a container a security risk?

Show answer

The Docker socket gives full control over the Docker daemon — meaning the container can create new containers, mount the host filesystem, run privileged containers, etc. Anyone with access to the socket has effective root on the host. A compromised container with socket access = host compromise. Avoid it; use the Docker API over TLS with client certs if remote access is needed.

Q4. A container works locally but crashes in production with "permission denied". What do you check?

Show answer

1. Check USER in Dockerfile — if running as non-root, the app may lack permissions to write to mounted volumes (volume UID/GID mismatch). 2. Check if production uses --read-only flag. 3. Check if the app tries to bind to port <1024 (requires root or CAP_NET_BIND_SERVICE). 4. Check SELinux/AppArmor on the production host — they may block operations allowed locally. 5. docker exec container id to see what user is actually running.
