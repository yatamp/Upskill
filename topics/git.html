Git & Branching Strategies

Level: Beginner → Intermediate  ·  Estimated: 2–3 days of reading + practice

What is Git and Why It Matters

Git is a distributed version control system. Every developer has a full copy of the repo — including its entire history — on their machine. There is no single "lock" on files; multiple people can work in parallel.


Key things Git tracks:

  • Snapshots, not diffs — each commit stores a snapshot of every file (using content-addressable storage via SHA-1/SHA-256 hashes).
  • The object model — blob (file content), tree (directory), commit (snapshot + metadata), tag.
  • Refs — branches and tags are just pointers (files containing a commit hash) stored in .git/refs/.
  • HEAD — a pointer to the current branch or commit you have checked out.

How Git Works Internally (the part most people miss)

Understanding this makes every Git command click.

Object Store

Everything in Git is an object (blob/tree/commit/tag) stored in .git/objects/, named by its SHA hash. Git is fundamentally a content-addressed filesystem.

The Three Areas

Working tree → your files on disk. Index / staging area → what will go into the next commit. Repository → committed history.

Branches are cheap

A branch is just a 41-byte file (a SHA hash). Creating a branch is O(1) — that's why branching in Git is fast and encouraged.

Merge vs Rebase

Merge creates a merge commit preserving history. Rebase replays commits on top of a new base, creating a linear history but rewriting commit SHAs.

Fast-forward

If the current branch is a direct ancestor of the target, Git just moves the pointer — no merge commit needed.

Remote refs

origin/main is a remote-tracking branch — a local copy of what the remote looked like when you last fetched. git fetch updates it; git pull fetches + merges.

Branching Strategies

Each strategy is a social contract for how your team uses branches. No one is universally best — the right choice depends on team size, release cadence, and deployment model.


Feature Branches

Each feature or bug lives on its own branch. Merged via PR when ready. Good default for most teams. Risk: long-lived branches diverge.

Git Flow

main (production) + develop (integration) + feature/release/hotfix branches. Structured, predictable. Can be heavy for fast-moving teams.

GitHub Flow

Single main branch. Feature branches PR directly into main. Simple, fast. Requires CI + feature flags if you deploy from main continuously.

Trunk-Based Development

Everyone integrates directly into main (or very short-lived branches, <1 day). Maximizes integration frequency. Requires strong CI and feature flags. Preferred by Google, Meta.

Release Branches

Cut a release branch when stabilizing for release. Hotfixes go into release branch AND back-merged to main. Common in software with multiple live versions.

Environment Branches

Branches named after environments (dev, staging, prod). Pushes trigger deploys. Simple but can get messy — usually better to use pipelines + tags instead.

Pull Request (PR) Workflow

PRs are the gate between a feature branch and the main line. A good PR workflow includes:

  • Small, focused PRs — easier to review, less conflict risk. Aim for <400 lines.
  • Descriptive title + description — what does this change and why.
  • CI checks pass before requesting review (tests, linting, security scan).
  • At least one reviewer — code review catches bugs and shares knowledge.
  • Squash or merge strategy — agree as a team. Squash = clean linear history. Merge commit = full context preserved.
  • Branch protection rules — require PR + passing CI before merging to main.

Hands-On Project: Simulate a Team Workflow

Do this on your local machine. No GitHub account needed for steps 1–5.

1
Set up a bare repo as a "remote" 
mkdir ~/git-practice && cd ~/git-practice
git init --bare remote.git
git clone remote.git alice
git clone remote.git bob

You now have two local "developer" clones sharing the same remote.

2
Alice creates a feature branch and makes commits 
cd ~/git-practice/alice
echo "# My Project" > README.md
git add README.md && git commit -m "init: add README"
git push origin main

git switch -c feature/login
echo "login logic here" > login.py
git add login.py && git commit -m "feat: add login module"
git push origin feature/login
3
Bob makes a conflicting change on main 
cd ~/git-practice/bob
git pull origin main
echo "logout logic" > login.py
git add login.py && git commit -m "feat: add logout"
git push origin main
4
Alice rebases her branch on top of main — resolve the conflict 
cd ~/git-practice/alice
git fetch origin
git rebase origin/main
# Git will pause at the conflict — edit login.py to merge both changes
git add login.py
git rebase --continue
git push origin feature/login --force-with-lease
5
Simulate a PR merge + tag a release 
cd ~/git-practice/alice
git switch main && git pull origin main
git merge --no-ff feature/login -m "merge: feature/login -> main"
git push origin main
git tag v1.0.0 -m "First release"
git push origin v1.0.0
6
Explore the object store 
ls .git/objects/        # see the SHA files
git cat-file -p HEAD    # inspect the commit object
git cat-file -p HEAD^{tree}  # inspect the tree object
git reflog              # see every HEAD movement

Check Your Understanding

Q1. What is the difference between git fetch and git pull?

Show answer

git fetch downloads remote changes and updates your remote-tracking branches (e.g. origin/main) but does NOT touch your working branch. git pull = git fetch + git merge (or rebase, depending on config). Prefer fetch first so you can inspect what changed before merging.

Q2. When would you use git rebase instead of git merge, and what's the golden rule?

Show answer

Use rebase to keep a linear history on your feature branch before merging. Golden rule: never rebase commits that have been pushed to a shared branch — it rewrites history (new SHA hashes) and breaks other people's branches. Use --force-with-lease (not --force) when you must push a rebased branch.

Q3. What is git reflog and why is it useful?

Show answer

Reflog records every time HEAD moved — branch switches, resets, rebases, etc. It lets you recover "lost" commits. Even after a git reset --hard or accidental branch delete, you can find the old commit SHA in reflog and restore it.

Q4. Your team wants to deploy multiple times per day. Which branching strategy fits best and why?

Show answer

Trunk-Based Development (TBD). Features merge to main quickly (ideally daily), preventing long-lived branch divergence. Requires: strong CI, feature flags to hide incomplete features, and fast PR turnarounds. GitHub Flow also works at this cadence but TBD maximizes integration frequency.
