GitHub Actions Deep Dive

Level: Intermediate → Advanced  ·  Estimated: 3–5 days  ·  Focus: real workflows, not toy examples

How GitHub Actions Works

GitHub Actions is an event-driven CI/CD platform built into GitHub. The execution model:


Events

Triggers that start a workflow. push, pull_request, schedule, workflow_dispatch (manual), workflow_call (reusable), repository_dispatch (API-triggered), release events.

Workflows

YAML files in .github/workflows/. One file = one workflow. Multiple workflows can run in parallel. They don't share state — each gets a fresh runner.

Jobs

Units of work within a workflow. Run in parallel by default. Use needs: to chain them. Each job gets a fresh runner VM (or container).

Steps

Sequential commands within a job. Share the same runner filesystem and environment. Can be shell commands (run:) or reusable Actions (uses:).

Runners

VMs that execute jobs. GitHub-hosted: ubuntu-latest, windows-latest, macos-latest. Self-hosted: your own machines/containers — needed for private network access, custom tooling, cost control.

Actions

Reusable building blocks published to the Marketplace or your org. Defined as Docker container, JS, or composite. Pin by full SHA for security.

Workflow YAML — Full Anatomy

Every key explained:

name: CI Pipeline                        # Display name in GitHub UI

on:                                      # Events that trigger this workflow
  push:
    branches: [main, 'release/**']       # Only on these branches
    paths-ignore: ['**.md', 'docs/**']   # Skip if only docs changed
  pull_request:
    types: [opened, synchronize]         # PR events to react to
  schedule:
    - cron: '0 6 * * 1'                  # Every Monday 6am UTC
  workflow_dispatch:                     # Allow manual trigger from UI
    inputs:
      environment:
        description: 'Deploy target'
        required: true
        default: 'staging'

env:                                     # Workflow-level env vars (all jobs)
  NODE_VERSION: '20'
  REGISTRY: ghcr.io

jobs:
  test:                                  # Job ID (used in needs:)
    name: Run Tests                      # Display name
    runs-on: ubuntu-latest               # Runner type
    timeout-minutes: 15                  # Kill job if it hangs

    strategy:                            # Matrix: run job N times with different vars
      fail-fast: false                   # Don't cancel siblings if one fails
      matrix:
        node: [18, 20]
        os: [ubuntu-latest, windows-latest]

    steps:
      - name: Checkout code
        uses: actions/checkout@v4        # Always pin major version at minimum
        with:
          fetch-depth: 0                 # Full history (needed for git-based versioning)

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'npm'                   # Built-in cache for package managers

      - name: Install dependencies
        run: npm ci                      # ci = clean install (uses lockfile exactly)

      - name: Run tests
        run: npm test
        env:
          DATABASE_URL: ${{ secrets.TEST_DB_URL }}   # Inject secret as env var

      - name: Upload coverage
        if: always()                     # Run even if tests fail
        uses: actions/upload-artifact@v4
        with:
          name: coverage-${{ matrix.node }}
          path: coverage/

  build:
    name: Build & Push Image
    runs-on: ubuntu-latest
    needs: test                          # Only run if 'test' job succeeds
    if: github.ref == 'refs/heads/main'  # Conditional: only on main branch
    permissions:
      contents: read
      packages: write                    # Need this to push to GHCR

    outputs:                             # Pass data to downstream jobs
      image-tag: ${{ steps.meta.outputs.tags }}

    steps:
      - uses: actions/checkout@v4

      - name: Docker meta
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/myorg/myapp
          tags: |
            type=sha
            type=semver,pattern={{version}}

      - uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}    # Auto-provided token

      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha          # GitHub Actions cache for Docker layers
          cache-to: type=gha,mode=max

  deploy:
    name: Deploy to Staging
    needs: build
    runs-on: ubuntu-latest
    environment:                        # GitHub Environment (requires approval if configured)
      name: staging
      url: https://staging.example.com
    steps:
      - name: Deploy
        run: |
          echo "Deploying ${{ needs.build.outputs.image-tag }}"
          # kubectl set image deployment/app app=${{ needs.build.outputs.image-tag }}

Contexts, Expressions & Conditionals

github context

github.ref — current branch/tag ref
github.sha — commit SHA
github.actor — user who triggered
github.event_name — push/pr/etc
github.repository — org/repo

secrets context

secrets.MY_SECRET — never logged, masked in output. Set in repo Settings → Secrets. Environment secrets override repo secrets. Use GITHUB_TOKEN (auto-provided) for repo access.

env context

Access env vars: env.MY_VAR. Set at workflow, job, or step level. Step-level overrides job-level overrides workflow-level.

Conditional expressions

if: github.ref == 'refs/heads/main'
if: failure() — run if previous step failed
if: always() — run regardless
if: contains(github.event.head_commit.message, '[skip ci]')

Step outputs

Pass data between steps:
echo "tag=v1.2.3" >> $GITHUB_OUTPUT
Then access: ${{ steps.mystep.outputs.tag }}

Job outputs

Pass data between jobs:
Define: outputs: { key: "${{ steps.x.outputs.y }}" }
Consume: ${{ needs.myjob.outputs.key }}

Caching — The Biggest Speed Win

- name: Cache node_modules
  uses: actions/cache@v4
  with:
    path: ~/.npm                    # What to cache
    key: npm-${{ hashFiles('package-lock.json') }}   # Cache key
    restore-keys: |                 # Fallback keys (partial match)
      npm-

How caching works: On cache hit, restore contents before the step. On cache miss, run normally, then save at end of job. Key must change when deps change — hash the lockfile. Cache is per-branch then falls back to default branch.
Cache pitfalls: Max 10GB per repo. Caches expire after 7 days of no access. Don't cache build outputs (they have side effects) — only cache dependencies. Use restore-keys as a fallback when the exact key misses.

Reusable Workflows — DRY Pipelines

Define once, call from any repo. Critical for large orgs with many services sharing the same pipeline.

# .github/workflows/reusable-build.yml  (in a shared repo)
on:
  workflow_call:                    # Makes this callable
    inputs:
      image-name:
        required: true
        type: string
    secrets:
      REGISTRY_TOKEN:
        required: true
    outputs:
      image-tag:
        value: ${{ jobs.build.outputs.tag }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      tag: ${{ steps.meta.outputs.tags }}
    steps:
      # ... build steps using ${{ inputs.image-name }}
# In your service repo — call the shared workflow
jobs:
  build:
    uses: myorg/shared-workflows/.github/workflows/reusable-build.yml@main
    with:
      image-name: my-service
    secrets:
      REGISTRY_TOKEN: ${{ secrets.REGISTRY_TOKEN }}

Secrets & Security Best Practices

TopicBest Practice
Pinning ActionsUse full SHA: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 — protects against tag mutation attacks. Use Dependabot to auto-update.
GITHUB_TOKEN permissionsMinimize: set permissions: contents: read at top level, only add what each job needs.
Third-party ActionsOnly use Actions from verified publishers or pin to SHA. uses: some-random/action@main is dangerous.
Secrets in logsNever echo ${{ secrets.FOO }} — it gets masked but it's still bad practice. Pass via env vars, not inline.
Environment secretsUse Environments for production secrets + required reviewers. This adds a manual approval gate before deploy jobs.
OIDC / Keyless AuthUse OIDC to authenticate to cloud providers without storing long-lived credentials. AWS, GCP, Azure all support it.

Hands-On Project: Build a Real CI/CD Pipeline

Create a GitHub repo with a Node.js app and build a complete pipeline.

1
Create the repo and app 
mkdir gh-actions-lab && cd gh-actions-lab
git init && git branch -M main

# Simple Express app
npm init -y
npm install express
npm install --save-dev jest supertest

cat > index.js << 'EOF'
const express = require('express');
const app = express();
app.get('/health', (req, res) => res.json({ status: 'ok' }));
app.get('/hello', (req, res) => res.json({ message: 'Hello World' }));
module.exports = app;
EOF

cat > server.js << 'EOF'
const app = require('./index');
app.listen(3000, () => console.log('Server on port 3000'));
EOF

cat > index.test.js << 'EOF'
const request = require('supertest');
const app = require('./index');
test('health check returns 200', async () => {
  const res = await request(app).get('/health');
  expect(res.status).toBe(200);
  expect(res.body.status).toBe('ok');
});
EOF

# Add test script to package.json
npm pkg set scripts.test="jest"
2
Create the CI workflow 
mkdir -p .github/workflows

cat > .github/workflows/ci.yml << 'EOF'
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - name: Upload results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results-node${{ matrix.node-version }}
          path: coverage/
EOF

Add a linting job that runs in parallel 
# Append to .github/workflows/ci.yml
cat >> .github/workflows/ci.yml << 'EOF'

  lint:
    runs-on: ubuntu-latest

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

EOF

Check Your Understanding

Q1. Job A takes 5min, Job B takes 3min, Job C depends on both. Total time with default settings?

Show answer

A and B run in parallel (no needs dependency between them). C starts after both finish. Total = max(5, 3) + C's time = 5min + C.

Q2. Your workflow fails because a secret isn't available in a fork PR. Why and how do you handle it?

Show answer

GitHub doesn't pass repository secrets to workflows triggered by PRs from forks (security measure — anyone can fork and exfiltrate secrets). Solutions: (1) Use pull_request_target with care (dangerous — gives access to base repo code), (2) design workflows so tests don't need secrets, (3) have maintainers trigger tests manually via workflow_dispatch after reviewing fork PRs.

Q3. You have 20 microservices each with the same CI pipeline. How do you avoid maintaining 20 copies of identical YAML?

Show answer

Use a reusable workflow in a shared repository. Each service calls the shared workflow with uses: myorg/shared/.github/workflows/ci.yml@main passing inputs like service-name. One change in the shared workflow propagates to all 20 services.

Q4. Why should you pin Actions to a full SHA rather than a tag like @v4?

Show answer

Tags are mutable — a maintainer (or attacker who compromises the maintainer's account) can move the tag to point to different, malicious code. A full SHA is immutable — it always refers to exactly the code you reviewed. Use tools like Dependabot to auto-PR SHA updates.
