Level: Intermediate → Advanced  ·  Focus: concurrency, performance, production patterns you use at Yahoo

Why Go — The Design Philosophy

Compiled & statically typed

Compiles to a single native binary. No runtime, no JVM, no interpreter. Cross-compile with GOOS=linux GOARCH=amd64 go build. Binary includes everything — trivial to containerise.

Goroutines, not threads

Goroutines are multiplexed onto OS threads by the Go scheduler (M:N threading). Start cost ~2KB stack vs ~8MB for an OS thread. Can run millions concurrently. This is why Go handles high concurrency cheaply.

CSP concurrency model

"Don't communicate by sharing memory; share memory by communicating." Channels are first-class — pass data between goroutines safely without locks. Locks still exist but are the escape hatch.

Explicit error handling

No exceptions. Functions return (value, error). Forces you to handle every error at the call site. Verbose but predictable — you always know what can fail.

Interfaces — implicit

A type satisfies an interface if it has the required methods — no implements keyword needed. Enables loose coupling and easy testing via mock injection without heavy frameworks.

GC but fast

Go has a concurrent, tri-color mark-and-sweep GC. Pauses are typically <1ms. For latency-sensitive paths, understand escape analysis: keep allocations on the stack (not heap) to reduce GC pressure.

// Basic goroutine
go func() {

// Buffered vs unbuffered channels
ch := make(chan int)      // unbuffered: send blocks until receive
ch := make(chan int, 100) // buffered: send blocks only when full

// Fan-out / Fan-in pattern — common in microservices
func fanOut(jobs <-chan Job, workers int) []<-chan Result {
    results := make([]<-chan Result, workers)
    for i := range workers {
        results[i] = worker(jobs)
    }
    return results
}

func merge(channels ...<-chan Result) <-chan Result {
    out := make(chan Result)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan Result) {
            defer wg.Done()
            for r := range c { out <- r }
        }(ch)
    }
    go func() { wg.Wait(); close(out) }()
    return out
}

// Context for cancellation — use in every long-running operation
func processKafkaMessage(ctx context.Context, msg Message) error {
    select {
    case <-ctx.Done():
        return ctx.Err() // cancelled or timed out
    default:
    }

    // Do work with deadline awareness
    ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
    defer cancel()

    return callDownstreamService(ctx, msg)
}

// WaitGroup — wait for N goroutines
var wg sync.WaitGroup
for _, item := range items {
    wg.Add(1)
    go func(i Item) {
        defer wg.Done()
        process(i)
    }(item) // capture loop var
}
wg.Wait()

// errgroup — WaitGroup + first error propagation
g, ctx := errgroup.WithContext(context.Background())
for _, url := range urls {
    url := url // capture
    g.Go(func() error {
        return fetch(ctx, url)
    })
}
if err := g.Wait(); err != nil {
    log.Fatal(err)
}

Interfaces & Dependency Injection

// Define interface for what you need, not what you have
type MessageStore interface {
    Save(ctx context.Context, msg Message) error
    Get(ctx context.Context, id string) (Message, error)
    Delete(ctx context.Context, id string) error
}

// Production implementation
type PostgresStore struct {
    db *pgxpool.Pool
}
func (s *PostgresStore) Save(ctx context.Context, msg Message) error { /* ... */ }

// Test mock — no frameworks needed
type mockStore struct {
    messages map[string]Message
}
func (m *mockStore) Save(_ context.Context, msg Message) error {
    m.messages[msg.ID] = msg
    return nil
}

// Service depends on interface, not concrete type
type MessageService struct {
    store  MessageStore   // inject interface
    cache  Cache
    logger *slog.Logger
}

func NewMessageService(store MessageStore, cache Cache, log *slog.Logger) *MessageService {
    return &MessageService{store: store, cache: cache, logger: log}
}

// In tests: inject mock. In prod: inject real impl.
svc := NewMessageService(&mockStore{messages: make(map[string]Message)}, nil, slog.Default())

Production Error Handling Patterns

// Sentinel errors — for known conditions
var (
    ErrNotFound     = errors.New("not found")
    ErrUnauthorized = errors.New("unauthorized")
)

// Custom error types — carry context
type ValidationError struct {
    Field   string
    Message string
}
func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed on %s: %s", e.Field, e.Message)
}

// Wrapping errors — preserve stack
if err := db.QueryRow(ctx, q, id).Scan(&msg); err != nil {
    if errors.Is(err, pgx.ErrNoRows) {
        return Message{}, fmt.Errorf("message %s: %w", id, ErrNotFound)
    }
    return Message{}, fmt.Errorf("querying message %s: %w", id, err)
}

// Unwrapping at the call site
msg, err := store.Get(ctx, id)
if errors.Is(err, ErrNotFound) {
    http.Error(w, "not found", http.StatusNotFound)
    return
}
var ve *ValidationError
if errors.As(err, &ve) {
    http.Error(w, ve.Message, http.StatusBadRequest)
    return
}

HTTP Server Patterns — Production Ready

// Minimal but production-grade HTTP server
func main() {
    mux := http.NewServeMux()

    // Register handlers
    mux.HandleFunc("GET /health", healthHandler)
    mux.HandleFunc("POST /api/v1/messages", createMessageHandler)
    mux.HandleFunc("GET /api/v1/messages/{id}", getMessageHandler)

    // Wrap with middleware chain
    handler := middleware.Chain(
        mux,
        middleware.RequestID,
        middleware.Logger(slog.Default()),
        middleware.Recover,
        middleware.Timeout(30*time.Second),
    )

    srv := &http.Server{
        Addr:         ":8080",
        Handler:      handler,
        ReadTimeout:  10 * time.Second,
        WriteTimeout: 30 * time.Second,
        IdleTimeout:  120 * time.Second,
    }

    // Graceful shutdown
    ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    go func() {
        if err := srv.ListenAndServe(); !errors.Is(err, http.ErrServerClosed) {
            slog.Error("server error", "err", err)
            os.Exit(1)
        }
    }()

    <-ctx.Done()
    slog.Info("shutting down")
    shutCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()
    srv.Shutdown(shutCtx)
}

// Middleware pattern
func Logger(log *slog.Logger) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()
            rw := &responseWriter{ResponseWriter: w, code: http.StatusOK}
            next.ServeHTTP(rw, r)
            log.Info("request",
                "method", r.Method,
                "path", r.URL.Path,
                "status", rw.code,
                "duration", time.Since(start),
                "request_id", r.Header.Get("X-Request-ID"),
            )
        })
    }
}

Performance — What to Actually Measure

TechniqueWhen to use
go test -bench=. -benchmemBenchmark hot paths. Look at allocs/op — each allocation is GC pressure.
go tool pprofCPU profile: find where time is spent. Heap profile: find what's allocating. Add net/http/pprof to your server for live profiling.
Sync.PoolReuse allocations for frequently created/destroyed objects (buffers, structs). Reduces GC pressure in hot paths.
Escape analysisgo build -gcflags='-m' shows what escapes to heap. Keep hot objects on stack — pass by value for small structs.
Pre-allocate slicesmake([]T, 0, expectedLen) — avoids repeated re-allocations as the slice grows.
strings.BuilderFor string concatenation in loops — much cheaper than + which allocates each time.
goroutine poolFor high-volume workloads, bound concurrency with a semaphore: sem := make(chan struct{}, maxWorkers)

sem := make(chan struct{}, 100) // max 100 concurrent
for msg := range messages {
    sem <- struct{}{}    // acquire
    go func(m Message) {
        defer func() { <-sem }() // release
        process(m)
    }(msg)
}

Testing Patterns

// Table-driven tests — Go standard
func TestValidateMessage(t *testing.T) {
    tests := []struct {
        name    string
        input   Message
        wantErr bool
    }{
        {"valid message", Message{ID: "1", Body: "hello"}, false},
        {"empty body",    Message{ID: "1", Body: ""},      true},
        {"no ID",         Message{ID: "", Body: "hello"},  true},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            err := ValidateMessage(tt.input)
            if (err != nil) != tt.wantErr {
                t.Errorf("got err=%v, wantErr=%v", err, tt.wantErr)
            }
        })
    }
}

Check Your Understanding

Q1. What's the difference between a goroutine leak and a goroutine panic? How do you detect each?

Show answer

A goroutine leak is a goroutine that starts but never exits — it holds resources indefinitely. Common cause: sending to a channel with no receiver, or blocking on a channel that never closes. Detect with runtime.NumGoroutine() in a pprof goroutine profile — count grows over time. A goroutine panic is an unrecovered panic in a goroutine — it crashes the entire program. Always add defer func() { if r := recover(); r != nil { log.Error(...) } }() at the top of goroutines you don't control.

Q2. When would you use sync.Mutex vs a channel?

Show answer

Use channels when transferring ownership of data between goroutines — the goroutine that sends gives up the data. Use Mutex when multiple goroutines need concurrent access to shared state (a cache, a counter, a map). Rule of thumb: if you're doing "worker receives tasks and produces results" → channel. If you're doing "multiple goroutines read/write the same cache" → Mutex + RWMutex for read-heavy cases.

Q3. Your service handles 300K Kafka messages/day but memory keeps growing. Where do you look first?

Show answer

1. pprof heap profile — find what's allocating on the heap. 2. Check for goroutine leaks — goroutines that started on message receipt but never exited. 3. Check slice/map growth — pre-allocate or cap them. 4. Look for closures capturing large variables. 5. Check if you're holding references to message payloads longer than needed. 6. Look for missing close() on channels or DB connections being opened without defer close.
