Apache Kafka

Level: Intermediate → Advanced  ·  You process 300K+ events/day — understand every moving part

How Kafka Works — The Architecture

Producers → [Topic: orders] → Consumers (Consumer Group A)
                ↓
      Partition 0 [msg1][msg2][msg3]... → Consumer 1
      Partition 1 [msg4][msg5][msg6]... → Consumer 2
      Partition 2 [msg7][msg8][msg9]... → Consumer 3
                ↓
         Broker 1 (leader p0, replica p1)
         Broker 2 (leader p1, replica p2)
         Broker 3 (leader p2, replica p0)

Topic & Partitions

A topic is split into partitions. Each partition is an ordered, immutable log. Messages are appended only — never updated. Each message has an offset (sequential ID within a partition).

Brokers & Replication

A cluster of brokers stores partitions. Each partition has one leader and N-1 replicas. Producers/consumers always talk to the leader. Replicas stay in sync → ISR (In-Sync Replicas). If leader dies, a new one is elected from ISR.

Consumer Groups

Each consumer group independently reads the topic. Within a group, each partition is assigned to exactly one consumer. Scaling consumers = more parallelism, but max consumers = number of partitions.

Offsets

Each consumer tracks its position (offset) per partition. Committed to __consumer_offsets topic. On restart, consumer resumes from last committed offset. This is how at-least-once delivery works.

Retention

Messages are kept for a configured time (e.g. 7 days) or size. Unlike a queue, Kafka doesn't delete on consume. Multiple consumer groups can read the same data independently — even replay from offset 0.

ZooKeeper → KRaft

Old Kafka used ZooKeeper for metadata. Kafka 3.x+ uses KRaft (Kafka Raft) — metadata stored in Kafka itself. Simpler, fewer moving parts. New deployments should use KRaft mode.

Producer Internals — What Actually Happens When You Send

Partitioning

If a key is provided: partition = hash(key) % numPartitions. Same key → always same partition → ordering guaranteed for that key. No key → round-robin (default) or sticky.

Batching

Producer batches messages before sending. batch.size = max bytes per batch. linger.ms = wait time to fill batch. Bigger batches = higher throughput but more latency. Tune both for your workload.

acks setting

acks=0: fire and forget (fastest, data loss possible). acks=1: leader ack only (leader crash = data loss). acks=all: all ISR must ack (safest, use in production with min.insync.replicas=2).

Idempotent producer

enable.idempotence=true — Kafka assigns a producer ID + sequence number. Broker deduplicates retries. Enables exactly-once semantics within a producer session.

Compression

compression.type=lz4 or snappy. Reduces network + disk I/O. LZ4 = best speed. Zstd = best ratio. Decompress on consumer side. Almost always worth enabling.

Transactions

Atomic produce across multiple topics. Use for exactly-once: read-process-write where write goes to Kafka. Set transactional.id. Kafka guarantees the entire transaction commits or rolls back.

Consumer Patterns in Go

// Production Kafka consumer in Go using confluent-kafka-go
package main

import (
    "context"
    "encoding/json"
    "log/slog"
    "os/signal"
    "syscall"
    "time"

    "github.com/confluentinc/confluent-kafka-go/v2/kafka"
)

type OrderEvent struct {
    ID     string  `json:"id"`
    Amount float64 `json:"amount"`
    UserID string  `json:"user_id"`
}

type Consumer struct {
    consumer  *kafka.Consumer
    processor EventProcessor
    log       *slog.Logger
}

func NewConsumer(brokers, groupID string, topics []string, p EventProcessor) (*Consumer, error) {
    c, err := kafka.NewConsumer(&kafka.ConfigMap{
        "bootstrap.servers":        brokers,
        "group.id":                 groupID,
        "auto.offset.reset":        "earliest",
        "enable.auto.commit":       false,  // manual commit — control exactly when
        "max.poll.interval.ms":     300000,
        "session.timeout.ms":       30000,
        "fetch.min.bytes":          1024,   // batch at least 1KB before delivering
        "fetch.wait.max.ms":        500,
    })
    if err != nil {
        return nil, err
    }
    c.SubscribeTopics(topics, nil)
    return &Consumer{consumer: c, processor: p}, nil
}

func (c *Consumer) Run(ctx context.Context) error {
    for {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
        }

        msg, err := c.consumer.ReadMessage(100 * time.Millisecond)
        if err != nil {
            if err.(kafka.Error).IsTimeout() {
                continue  // no messages, try again
            }
            c.log.Error("kafka read error", "err", err)
            continue
        }

        // Process message
        if err := c.processMessage(ctx, msg); err != nil {
            c.log.Error("processing failed",
                "topic", *msg.TopicPartition.Topic,
                "offset", msg.TopicPartition.Offset,
                "err", err,
            )
            // Decision: send to DLQ or retry?
            c.sendToDLQ(ctx, msg, err)
        }

        // Commit AFTER successful processing (at-least-once)
        if _, err := c.consumer.CommitMessage(msg); err != nil {
            c.log.Error("commit failed", "err", err)
        }
    }
}

func (c *Consumer) processMessage(ctx context.Context, msg *kafka.Message) error {
    var event OrderEvent
    if err := json.Unmarshal(msg.Value, &event); err != nil {
        return fmt.Errorf("unmarshal: %w", err)  // don't retry on bad JSON
    }
    return c.processor.Process(ctx, event)
}

Critical Config Settings

SettingWhat it controlsProduction recommendation
replication.factorHow many brokers hold a copy of each partition3 in production — tolerates 1 broker failure
min.insync.replicasMin replicas that must ack a write2 — combined with acks=all prevents data loss
retention.msHow long messages are kept7 days default — tune per topic data volume
max.message.bytesMax message sizeDefault 1MB — increase if you send large payloads, but prefer smaller messages
num.partitionsParallelism ceilingStart with 3-6 per topic — you can increase but never decrease
consumer: max.poll.recordsRecords returned per pollLower (e.g. 100) if processing is slow to avoid rebalance timeouts
consumer: session.timeout.msHow long before broker considers consumer dead30s — lower means faster rebalance detection

Common Problems & How to Debug

ProblemCauseFix
Consumer lag growingConsumers are slower than producersAdd more partitions + consumers. Profile the processing code. Use async processing with bounded goroutine pool.
Frequent rebalancesConsumer processing takes longer than session.timeout.msIncrease max.poll.interval.ms. Reduce max.poll.records. Move slow work to async goroutines, commit after.
Duplicate messagesConsumer crash after processing but before commitThis is at-least-once by design. Make processing idempotent — use message ID as dedup key in DB/Redis.
Messages out of orderMultiple partitions or no partition keyKafka only guarantees order within a partition. Use a consistent key (e.g. user_id) if order matters.
Producer data lossacks=1 and leader crash during writeSet acks=all + min.insync.replicas=2. Enable idempotent producer.

Check Your Understanding

Q1. Your orders topic has 6 partitions and your consumer group has 8 consumers. What happens?

Show answer

Only 6 consumers will be active — each assigned one partition. The other 2 will be idle. Max parallelism = number of partitions. To use all 8 consumers you'd need at least 8 partitions. This is why you should plan partition count ahead of time — you can increase but never decrease without downtime.

Q2. Your service must process payment events exactly once. How do you achieve this with Kafka?

Show answer

Kafka's exactly-once requires: (1) Idempotent producer (enable.idempotence=true) — deduplicates retries. (2) Transactional API — atomic read-process-write. (3) Or: at-least-once delivery + idempotent consumer — store the message offset/ID in your DB in the same transaction as the processing effect. The last approach is most common and practical.

Q3. When would you use a partition key vs no key?

Show answer

Use a key when: ordering matters for a specific entity (all events for user_id=123 must be processed in order). The key ensures all messages for that entity land in the same partition. Use no key (round-robin) when: order doesn't matter and you want even load distribution across partitions. Warning: if your keyspace is small (e.g. country codes), you'll get hot partitions — some consumers work much harder than others.
