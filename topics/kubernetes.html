Kubernetes

Level: Intermediate  ·  Estimated: 5–7 days  ·  Industry standard for container orchestration

Kubernetes Architecture

┌──────────────── Control Plane ─────────────────┐
│ API Server ←── kubectl / all communication │
│ etcd ── cluster state (key-value DB) │
│ Scheduler ── assigns pods to nodes │
│ Controller ── reconciliation loops │
└────────────────────────────────────────────────┘
          ↕ (API calls)
┌──── Worker Node ────┐ ┌──── Worker Node ────┐
│ kubelet │ │ kubelet │
│ kube-proxy │ │ kube-proxy │
│ container runtime │ │ container runtime │
│ [Pod][Pod][Pod] │ │ [Pod][Pod] │
└─────────────────────┘ └─────────────────────┘

API Server

The front door to everything. All kubectl commands, all internal component communication goes through it. Validates objects, persists to etcd, serves the REST API.

etcd

The cluster's source of truth — a distributed key-value store. Holds every Kubernetes object. If etcd dies without backup, you lose your cluster config. Back it up.

Scheduler

Watches for unscheduled pods and assigns them to nodes based on: resource requests, node affinity, taints/tolerations, pod anti-affinity rules.

Controllers

Reconciliation loops — continuously compare desired state vs actual state and act to close the gap. ReplicaSet controller ensures N pods are running. Deployment controller manages rolling updates.

kubelet

Agent on every node. Receives pod specs from API Server, tells the container runtime to start/stop containers, reports node and pod status back.

kube-proxy

Maintains network rules on nodes. Implements Service load balancing using iptables or IPVS. Routes traffic to the right pod based on Service selectors.

Core Objects — The Building Blocks

Pod

The smallest deployable unit. One or more containers sharing network namespace and storage. Containers in a pod communicate via localhost. Pods are ephemeral — don't manage them directly.

Deployment

Manages a ReplicaSet to maintain N pod replicas. Handles rolling updates and rollbacks. This is how you deploy stateless apps. Wraps: Deployment → ReplicaSet → Pod.

Service

Stable network endpoint for a set of pods. Types: ClusterIP (internal), NodePort (expose on each node), LoadBalancer (cloud LB), ExternalName (DNS alias). Uses label selectors to find pods.

Ingress

L7 HTTP routing rules — route by hostname/path to different Services. Requires an Ingress Controller (nginx-ingress, Traefik, ALB controller). Handles TLS termination.

ConfigMap

Non-sensitive configuration as key-value pairs. Mount as volume (files in container) or inject as env vars. Changes require pod restart to take effect (unless using hot-reload).

Secret

Like ConfigMap but for sensitive data. Base64 encoded (not encrypted by default — use encryption at rest + external secrets operator for production). Never commit to git.

StatefulSet

Like Deployment but for stateful apps (databases, queues). Provides: stable network names (pod-0, pod-1), stable storage (PVC per pod), ordered deployment/scaling.

DaemonSet

Runs one pod per node. Used for: log collectors (Fluentd), monitoring agents (Node Exporter), network plugins. Automatically adds pod to new nodes.

Job / CronJob

Job: run a pod to completion (batch tasks, migrations). CronJob: run a Job on a schedule. Like cron but in Kubernetes.

Namespace

Virtual cluster within a cluster. Scope for resource names, RBAC, network policies, resource quotas. Use to isolate: dev/staging/prod or different teams.

PersistentVolume

Storage provisioned in the cluster (EBS, NFS, etc.). PVC (PersistentVolumeClaim) is a pod's request for storage. StorageClass enables dynamic provisioning.

HorizontalPodAutoscaler

Automatically scales Deployment replicas based on CPU/memory/custom metrics. Requires metrics-server. Essential for handling traffic spikes.

Writing Production Manifests

Complete Deployment with all important fields:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
    version: v2.1.3
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp              # Must match template labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1       # At most 1 pod down during update
      maxSurge: 1             # At most 1 extra pod during update
  template:
    metadata:
      labels:
        app: myapp
        version: v2.1.3
    spec:
      serviceAccountName: myapp-sa   # Least-privilege service account

      # Spread pods across nodes for HA
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: myapp

      containers:
        - name: app
          image: myregistry/myapp:v2.1.3   # Always pin — never 'latest' in prod
          ports:
            - containerPort: 8080

          # Resource requests & limits — CRITICAL
          resources:
            requests:
              cpu: "100m"        # 0.1 CPU core
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"

          # Liveness: if this fails, container is restarted
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15    # Wait before first check
            periodSeconds: 20
            failureThreshold: 3

          # Readiness: if this fails, pod removed from Service endpoints
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5

          # Startup: disables liveness until app has started
          startupProbe:
            httpGet:
              path: /health
              port: 8080
            failureThreshold: 30
            periodSeconds: 10

          env:
            - name: NODE_ENV
              value: production
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: myapp-secrets
                  key: db-password
            - name: APP_CONFIG
              valueFrom:
                configMapKeyRef:
                  name: myapp-config
                  key: config.json

          # Mount a ConfigMap as files
          volumeMounts:
            - name: config
              mountPath: /app/config
              readOnly: true
            - name: tmp
              mountPath: /tmp          # Writable temp dir when using read-only root

          # Security hardening
          securityContext:
            runAsNonRoot: true
            runAsUser: 1000
            readOnlyRootFilesystem: true
            allowPrivilegeEscalation: false
            capabilities:
              drop: ["ALL"]

      volumes:
        - name: config
          configMap:
            name: myapp-config
        - name: tmp
          emptyDir: {}

      # Graceful shutdown
      terminationGracePeriodSeconds: 60

Service + Ingress:

---
apiVersion: v1
kind: Service
metadata:
  name: myapp
  namespace: production
spec:
  selector:
    app: myapp               # Routes to pods with this label
  ports:
    - port: 80               # Service port
      targetPort: 8080       # Pod port
  type: ClusterIP            # Internal only — use Ingress for external

---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp
  namespace: production
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
    - hosts: [app.example.com]
      secretName: myapp-tls
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: myapp
                port:
                  number: 80

Autoscaling & Resource Management

---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # Scale up when avg CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80

Resource requests matter: Without requests defined, the scheduler places pods randomly. Without limits, one bad pod can eat all node memory. Set both. A pod with no limits will be OOMKilled on a busy node. A pod with no requests will be evicted first during resource pressure.

Debugging Guide — When Things Go Wrong

SymptomDebug steps
Pod stuck in Pendingkubectl describe pod → Events section. Usually: insufficient CPU/memory on nodes, no node matches affinity, PVC not bound.
Pod in CrashLoopBackOffkubectl logs pod --previous → app crash reason. Check: CMD/ENTRYPOINT valid? Env vars set? Health check endpoint exists?
Pod in ImagePullBackOffImage name typo? Registry auth missing? kubectl describe pod shows exact pull error. Check imagePullSecrets.
Pod Running but not reachableIs it Ready? Check readiness probe. kubectl get endpoints myservice — is the pod's IP listed? If no endpoints, label selector mismatch.
Service not routingkubectl get endpoints svc/myapp — should list pod IPs. If empty: label selector doesn't match pod labels. kubectl port-forward pod/mypod 8080:8080 to bypass service and test pod directly.
OOMKilledPod killed by kernel for exceeding memory limit. kubectl describe pod shows "OOMKilled". Increase memory limit or fix memory leak in app.
Rolling update stuckkubectl rollout status deploy/myapp. New pods may not pass readiness. Check new version's health endpoint. kubectl rollout undo deploy/myapp to revert.

Check Your Understanding

Q1. What's the difference between a liveness probe, readiness probe, and startup probe?

Show answer

Liveness: Is the container alive? If it fails, Kubernetes restarts the container. Use for detecting deadlocks.
Readiness: Is the container ready to serve traffic? If it fails, pod is removed from Service endpoints but NOT restarted. Use for temporary unavailability (loading cache, warming up).
Startup: Has the app finished starting? Disables liveness/readiness until it passes. Use for slow-starting apps to avoid being killed before they're ready.

Q2. A pod is Running but getting no traffic. How do you debug?

Show answer

1. kubectl get endpoints svc/myapp — if empty, no pods match the selector. Compare service spec.selector with pod labels. 2. If endpoints exist, check readiness probe — pod may be Running but not Ready. kubectl describe pod → check ready status and probe events. 3. kubectl port-forward pod/mypod 8080:8080 — bypass the service to test pod directly. If pod works direct but not via service, it's a service/kube-proxy issue.

Q3. Why should you never use latest image tag in production?

Show answer

Tags are mutable — latest today is a different image than latest tomorrow. Problems: (1) You can't rollback to "the previous version" reliably. (2) Different nodes may pull different versions if they pull at different times. (3) You lose reproducibility — you don't know what's actually running. Always pin to a specific version tag or SHA256 digest (myapp@sha256:abc123).

Q4. When would you use a StatefulSet instead of a Deployment?

Show answer

StatefulSet when: (1) Each pod needs a stable, persistent identity (pod-0, pod-1 — never random names). (2) Each pod needs its own persistent storage that stays with the pod through restarts. (3) Pods need ordered, sequential startup/shutdown (e.g., primary before replicas). Use cases: databases (Postgres, MySQL, Cassandra), message queues (Kafka, RabbitMQ), distributed coordination systems (etcd, ZooKeeper).
