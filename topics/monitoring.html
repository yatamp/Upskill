Monitoring & Observability

Level: Intermediate → Advanced  ·  Prometheus · Grafana · Datadog · Structured Logging · Alerting

The Three Pillars of Observability

Metrics

Numeric measurements over time. Low cardinality, cheap to store. Tell you WHAT is wrong at a high level. Examples: request rate, error rate, latency p99, CPU%, memory, Kafka consumer lag.

Logs

Timestamped records of discrete events. High cardinality, expensive to store/query. Tell you WHAT happened and the context. Structured logs (JSON) >> plain text for querying.

Traces

A request's journey through distributed services. Shows WHERE time was spent across service boundaries. Essential for microservices. Tools: Jaeger, Zipkin, Datadog APM, AWS X-Ray.

The workflow

Alert fires on a metric → jump to dashboard for context → correlate with logs to find the specific error → traces show which downstream service is slow. Each pillar answers a different question.

Prometheus — Metrics Collection

Prometheus is a pull-based metrics system. It scrapes HTTP endpoints (/metrics) on a schedule and stores time-series data.


4 Metric Types

Counter: only goes up (requests_total, errors_total). Gauge: up/down (memory_bytes, queue_depth). Histogram: bucketed distribution (request_duration_seconds). Summary: pre-calculated quantiles (use histogram instead — more flexible).

Labels

Key-value pairs on metrics. http_requests_total{method="GET", status="200", service="auth"}. Labels enable slicing — aggregate by service, endpoint, status. Warning: high-cardinality labels (user_id, request_id) blow up storage.

PromQL

Prometheus Query Language. Powers Grafana dashboards and alerts. rate(http_requests_total[5m]) = per-second request rate. histogram_quantile(0.99, rate(...)) = p99 latency.

Service Discovery

In Kubernetes: Prometheus auto-discovers pods/services via annotations. Add prometheus.io/scrape: "true" and prometheus.io/port: "8080" to your pod spec. No manual config per service.


Instrument your Go service:

import (
    "github.com/prometheus/client_golang/prometheus"
    "github.com/prometheus/client_golang/prometheus/promauto"
    "github.com/prometheus/client_golang/prometheus/promhttp"
)

var (
    // Counter — requests processed
    requestsTotal = promauto.NewCounterVec(
        prometheus.CounterOpts{
            Name: "http_requests_total",
            Help: "Total HTTP requests",
        },
        []string{"method", "path", "status"},
    )

    // Histogram — request duration with buckets
    requestDuration = promauto.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "http_request_duration_seconds",
            Help:    "HTTP request duration",
            Buckets: []float64{.005, .01, .025, .05, .1, .25, .5, 1, 2.5},
        },
        []string{"method", "path"},
    )

    // Gauge — current Kafka consumer lag
    kafkaLag = promauto.NewGaugeVec(
        prometheus.GaugeOpts{
            Name: "kafka_consumer_lag",
            Help: "Current Kafka consumer lag by topic/partition",
        },
        []string{"topic", "partition"},
    )
)

// Middleware to instrument all HTTP handlers
func metricsMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        rw := &responseWriter{ResponseWriter: w, code: http.StatusOK}
        next.ServeHTTP(rw, r)
        dur := time.Since(start).Seconds()

        status := strconv.Itoa(rw.code)
        requestsTotal.WithLabelValues(r.Method, r.URL.Path, status).Inc()
        requestDuration.WithLabelValues(r.Method, r.URL.Path).Observe(dur)
    })
}

// Expose /metrics endpoint
mux.Handle("/metrics", promhttp.Handler())

Grafana — Dashboards

Grafana visualises data from Prometheus (and Datadog, Loki, CloudWatch, etc). Key things to know:

Essential PromQL queries for your services:

# Request rate (per second, 5-min window)
rate(http_requests_total{service="auth"}[5m])

# Error rate percentage
rate(http_requests_total{status=~"5.."}[5m])
  / rate(http_requests_total[5m]) * 100

# p99 latency
histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket{service="auth"}[5m])
)

# Kafka consumer lag sum across all partitions
sum by (topic) (kafka_consumer_lag{group="my-service"})

# CPU usage by pod
rate(container_cpu_usage_seconds_total{namespace="production"}[5m]) * 100

# Memory usage
container_memory_working_set_bytes{namespace="production"} / 1024 / 1024

Datadog — Enterprise Observability

Datadog is what you use at Yahoo. It's a unified platform: metrics, logs, traces, APM, synthetics, dashboards, alerting.


DogStatsD

Metrics agent — your app sends metrics via UDP to the local Datadog agent, which batches and forwards to Datadog. Library: github.com/DataDog/datadog-go/v5/statsd.

APM / Distributed Tracing

Auto-instrument with dd-trace-go. Get traces across Go services with zero manual instrumentation. Correlates traces → logs → metrics in one click.

Log Management

Ship structured JSON logs → Datadog auto-parses fields. Use dd.trace_id and dd.span_id fields to link logs to traces. Log attributes become filterable facets.

Monitors & Alerts

Set thresholds on any metric, log query, or APM trace. Alert on: error rate > 1%, p99 latency > 500ms, Kafka lag > 10K, pod OOMKill rate. Route to PagerDuty for critical, Slack for warn.


// Instrument a Go service with Datadog
import (
    "gopkg.in/DataDog/dd-trace-go.v1/ddtrace/tracer"
    "gopkg.in/DataDog/dd-trace-go.v1/contrib/net/http"
    ddstatsd "github.com/DataDog/datadog-go/v5/statsd"
)

func main() {
    // Start tracer
    tracer.Start(
        tracer.WithServiceName("my-service"),
        tracer.WithEnv("production"),
        tracer.WithServiceVersion("v2.1.3"),
    )
    defer tracer.Stop()

    // Metrics client
    statsd, _ := ddstatsd.New("localhost:8125")
    defer statsd.Close()

    // Auto-instrumented HTTP router
    mux := httptrace.NewServeMux()
    mux.HandleFunc("/api/orders", func(w http.ResponseWriter, r *http.Request) {
        // Custom metric
        statsd.Incr("orders.processed", []string{"env:production"}, 1)
        statsd.Timing("orders.latency", time.Since(start), nil, 1)
    })
}

Structured Logging — The Right Way

// Use slog (Go 1.21+ standard library) — structured by default
import "log/slog"

// Set up JSON handler for production
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelInfo,
}))
slog.SetDefault(logger)

// Good: structured fields, not string formatting
slog.Info("order processed",
    "order_id", order.ID,
    "user_id", order.UserID,
    "amount", order.Amount,
    "duration_ms", time.Since(start).Milliseconds(),
)

// Bad: unstructured — can't query on order_id in Datadog
log.Printf("order processed: id=%s user=%s amount=%.2f", order.ID, order.UserID, order.Amount)

// Always include trace context for correlation
slog.InfoContext(ctx, "downstream call failed",
    "service", "payments",
    "error", err,
    "trace_id", trace.SpanFromContext(ctx).SpanContext().TraceID().String(),
)

// Output (JSON, Datadog auto-parses this):
// {"time":"2025-04-15T10:23:01Z","level":"INFO","msg":"order processed",
//  "order_id":"ord_123","user_id":"usr_456","amount":99.99,"duration_ms":12}

SLOs, SLIs & Error Budgets

TermDefinitionExample
SLIService Level Indicator — the metric you measure% of requests returning 2xx in <200ms
SLOService Level Objective — the target you commit to99.9% of requests succeed within 200ms, measured monthly
SLAService Level Agreement — contractual commitment with penaltyExternal customers get 99.9% uptime or credits
Error BudgetHow much failure you're allowed: 100% - SLO99.9% SLO = 0.1% budget = ~43 minutes/month downtime allowed
Burn rateHow fast you're consuming the error budgetBurn rate 10x = consuming monthly budget in 3 days → page immediately
Practical SLO alerting: Alert when your error budget burn rate is unsustainably high — not just when you breach the SLO (too late by then). Burn rate alerts are the Google SRE recommendation. Datadog has built-in SLO tracking.

Check Your Understanding

Q1. Your p99 latency alert fires but average latency looks fine. What could cause this and how do you investigate?

Show answer

p99 high + average normal = a small percentage of requests are very slow. Causes: (1) A specific slow downstream dependency (DB slow query, external API timeout). (2) A specific code path only some requests hit. (3) GC pauses. (4) Hot partition in Kafka (one consumer much slower). Investigation: use distributed tracing to find the slow spans in the slow requests. Filter Datadog APM by p99 latency to see which traces are outliers.

Q2. What's wrong with adding user_id as a Prometheus label?

Show answer

Cardinality explosion. Prometheus stores a separate time series for each unique label combination. With 1M users, adding user_id creates 1M+ time series per metric — this will OOM your Prometheus and make queries unusably slow. Labels should have low cardinality (typically <100 unique values): service, env, method, status code, region. Use Datadog logs or distributed traces (which can handle high cardinality) when you need to query by user_id.
