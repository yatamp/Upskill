Networking Basics

Level: Beginner → Intermediate  ·  Estimated: 3–4 days  ·  Focus: the IP-level internals most people skip

The OSI Model — Why It Actually Matters

The OSI model is a framework for understanding how data travels from app to wire and back. Every tool you use (curl, DNS, TLS, load balancers) operates at a specific layer. Knowing the layer tells you where to look when something breaks.


LayerNameWhat it doesExamples
7ApplicationProtocols your apps speakHTTP, HTTPS, DNS, SMTP, SSH, gRPC
6PresentationEncoding, encryption, compressionTLS/SSL, JPEG, JSON serialization
5SessionManages sessions between appsRPC, NFS, SIP
4TransportEnd-to-end delivery, ports, reliabilityTCP, UDP
3NetworkRouting between networks, IP addressesIP, ICMP, routing protocols
2Data LinkNode-to-node delivery on same network, MAC addressesEthernet, Wi-Fi, ARP
1PhysicalBits on the wireCables, fiber, radio signals

Practical shortcut: Most DevOps work lives at L3 (IP routing, subnets, firewall rules) and L4 (TCP/UDP ports, load balancers). L7 is where your apps and HTTP work. L1/L2 is usually the network team's concern.

IP Addressing & Subnets (the bit everyone skips)

An IPv4 address is 32 bits written as 4 octets: 192.168.1.100. A subnet mask defines which bits are the network portion and which are the host portion.


CIDR Notation

192.168.1.0/24 means the first 24 bits are the network. Leaves 8 bits for hosts → 2⁸ = 256 addresses (254 usable, .0 = network, .255 = broadcast).

Private Ranges (RFC 1918)

10.0.0.0/8 — 16M addresses
172.16.0.0/12 — 1M addresses
192.168.0.0/16 — 65K addresses
These never route on the public internet.

Subnet Math

A /26 gives 64 addresses (62 usable). Each bit you add to the prefix halves the host count. /30 = 4 addresses = point-to-point link.

NAT

Network Address Translation — your router maps many private IPs to one public IP. Outgoing connections work; incoming need port forwarding. This is why your laptop has 192.168.x.x but the internet sees your router's IP.

IPv6

128-bit addresses: 2001:db8::1. Designed to replace IPv4. Dual-stack (both) is common today. Key difference: no NAT — every device gets a globally routable address.

Loopback & Special IPs

127.0.0.1 (localhost) — your machine talking to itself. 0.0.0.0 — listen on all interfaces. 169.254.x.x — APIPA (no DHCP found). 255.255.255.255 — broadcast to everyone.

DNS — How Names Become IPs

DNS is a distributed, hierarchical database. Every name lookup walks this tree until it finds an authoritative answer.


Full resolution chain for www.example.com:

Your app
  → OS resolver (checks /etc/hosts first, then cache)
    → Recursive resolver (your ISP or 8.8.8.8)
      → Root nameserver (knows who handles .com)
        → TLD nameserver (knows who handles example.com)
          → Authoritative nameserver (knows www → 93.184.216.34)
              ← TTL cached at each step on the way back

Record types you must know:

RecordPurposeExample
AHostname → IPv4 addresswww.example.com → 93.184.216.34
AAAAHostname → IPv6 addresswww.example.com → 2606:2800::1
CNAMEAlias → another hostnameblog.example.com → example.com
MXMail server for domainexample.com → mail.example.com
TXTArbitrary text (SPF, DKIM, ownership proofs)"v=spf1 include:..."
NSAuthoritative nameservers for domainexample.com → ns1.example.com
PTRReverse DNS: IP → hostname34.216.184.93.in-addr.arpa → www.example.com
SRVService discovery (host + port)Used by Kubernetes, SIP, XMPP

TTL matters in DevOps: When you change a DNS record, old IPs are still cached until TTL expires. Lower TTL before a migration; raise it again after. A 300s TTL = 5 minutes of propagation pain; a 86400s TTL = 24 hours.

HTTP & HTTPS — The Protocol Your Apps Speak

HTTP is a stateless, text-based request-response protocol. Every request is independent.


HTTP/1.1 vs HTTP/2 vs HTTP/3

1.1 = one request per TCP connection (head-of-line blocking). HTTP/2 = multiplexing (many streams one TCP connection). HTTP/3 = runs over QUIC (UDP-based), no TCP HOL blocking, faster connection setup.

Request anatomy

Method + URL + Version, then headers, then body.
GET /api/users HTTP/1.1
Host: api.example.com
Authorization: Bearer ...

Status codes

2xx = success. 3xx = redirect. 4xx = client error (401 unauth, 403 forbidden, 404 not found, 429 rate limit). 5xx = server error (500 internal, 502 bad gateway, 503 unavailable).

Important Headers

Content-Type, Authorization, Cache-Control, X-Request-ID (tracing), Retry-After, CORS headers.

TLS — How HTTPS Actually Works

TLS (Transport Layer Security) sits between TCP and HTTP. It provides: confidentiality (encryption), integrity (no tampering), and authentication (server proves its identity).


TLS 1.3 Handshake (simplified):

Client                          Server
  |------ ClientHello ------------>|  (supported ciphers, random, key share)
  |<----- ServerHello + Cert ------|  (chosen cipher, server's public key, certificate)
  |  [Client verifies cert against trusted CA]
  |------ Finished (encrypted) --->|  (session key derived from key exchange)
  |<----- Finished (encrypted) ----|
  |======= Encrypted HTTP =========|

Certificate Chain

Your cert is signed by an Intermediate CA, which is signed by a Root CA. Browsers trust Root CAs (pre-installed). A broken chain = SSL errors. Use openssl verify to debug.

SNI

Server Name Indication — lets the client send the hostname in the TLS handshake, so one server/IP can host multiple certs. This is how shared hosting works.

mTLS

Mutual TLS — both client AND server present certificates. Used in service meshes (Istio, Linkerd), zero-trust networks, and API authentication.

Certificate Management

Let's Encrypt = free, auto-renewing certs. Cert-manager = Kubernetes operator for auto-renewal. Watch expiry with monitoring — expired certs cause outages.

TCP vs UDP — The Transport Layer

PropertyTCPUDP
ConnectionConnection-oriented (3-way handshake: SYN → SYN-ACK → ACK)Connectionless — fire and forget
ReliabilityGuaranteed delivery, ordering, retransmissionBest-effort, no guarantees
SpeedSlower (overhead for guarantees)Faster (no handshake, no ACK wait)

TCP states to know: ESTABLISHED (active connection), TIME_WAIT (closed but waiting for stray packets — can exhaust ports under high load), CLOSE_WAIT (remote closed but local hasn't — usually a bug in your app).

Load Balancers

A load balancer distributes traffic across multiple backend servers. Two main types:


L4 (Network) Load Balancer

Operates on TCP/UDP. Routes by IP+port. Extremely fast — no packet inspection. Can't route based on URL paths or headers. Examples: AWS NLB, HAProxy (TCP mode).

L7 (Application) Load Balancer

Understands HTTP. Can route by URL path, hostname, headers, cookies. Terminates TLS. Slower than L4 but far more flexible. Examples: AWS ALB, nginx, Envoy, Traefik.

Algorithms

Round-robin, least-connections, IP-hash (sticky sessions), random, weighted. Least-connections is best when requests have variable processing time.

Health Checks

LB probes backends (HTTP 200 or TCP connect). Unhealthy backends are removed. Set: interval, timeout, healthy threshold, unhealthy threshold. Critical for zero-downtime deploys.

Connection Draining

When removing a backend, the LB stops new connections but lets existing ones finish. Prevents dropped requests during deploys. AWS: "deregistration delay" (300s default).

Sticky Sessions

Routes the same client to the same backend (via cookie or IP hash). Needed for stateful apps. Problem: uneven load if one user is heavy. Better: move state to Redis/DB.

Firewalls & Security Groups

Stateless Firewall

Evaluates each packet independently against rules (source IP, dest IP, port, protocol). Must allow both inbound and return traffic explicitly. Fast, simple.

Stateful Firewall

Tracks connection state. If you allow outbound TCP, return packets are automatically allowed. Most modern firewalls are stateful. AWS Security Groups are stateful.

AWS Security Groups

Virtual firewall at the instance level. Inbound rules default-deny. Outbound default-allow. Rules are allow-only (no explicit deny). Applied per-ENI.

Network ACLs

Subnet-level, stateless. Rules evaluated in order by number. Both allow AND deny rules. Applied to all traffic entering/leaving a subnet. Used for broad blocks.

Hands-On Project: Trace a Request End-to-End

Run these on your Mac terminal. Understand every output line.

1
DNS resolution chain 
# What's in your local DNS cache?
sudo dscacheutil -statistics

# Full DNS trace for a domain
dig +trace google.com

# What nameservers are authoritative?
dig NS google.com

# Reverse lookup an IP
dig -x 8.8.8.8
2
Trace the network path 
# See every hop to Google
traceroute -n google.com

# Better: continuous with packet loss stats
mtr --report google.com
3
Inspect TLS in detail 
# See full TLS handshake + certificate chain
openssl s_client -connect google.com:443 -showcerts

# Check cert expiry
echo | openssl s_client -connect google.com:443 2>/dev/null \
  | openssl x509 -noout -dates

# Test TLS version support
curl -v --tlsv1.2 https://google.com 2>&1 | grep "SSL connection"
4
Inspect HTTP headers 
# Full request/response headers
curl -v https://httpbin.org/get 2>&1

# Follow redirects, show all
curl -L -v https://google.com 2>&1 | grep -E "^[<>]"

# Check CORS headers
curl -H "Origin: http://example.com" -v https://httpbin.org/get 2>&1
5
Subnet math — calculate by hand then verify 
# Given 10.0.4.0/22 — how many hosts? What's the broadcast?
# /22 = 22 bits network, 10 bits host = 2^10 = 1024 addresses
# Broadcast = 10.0.7.255   (10.0.4.0 + 1023)
# Verify with:
python3 -c "import ipaddress; n=ipaddress.ip_network('10.0.4.0/22'); print(f'hosts={n.num_addresses-2}, broadcast={n.broadcast_address}')"
6
Capture and read real traffic with tcpdump 
# In terminal 1: capture DNS queries
sudo tcpdump -i any port 53 -n

# In terminal 2: trigger DNS queries
dig example.com
curl https://example.com

# Watch terminal 1 — you'll see the DNS query and response packets

Check Your Understanding

Q1. A server has IP 10.0.0.5 in subnet 10.0.0.0/28. How many usable host IPs are in this subnet? What's the broadcast address?

Show answer

/28 = 28 bits network, 4 bits host → 2⁴ = 16 addresses. Minus network (.0) and broadcast (.15) = 14 usable hosts. Broadcast = 10.0.0.15.

Q2. You change a DNS A record but users still hit the old IP for 24 hours. What's the cause and how do you prevent it next time?

Show answer

The old TTL was 86400 (24 hours) — recursive resolvers cached it. Prevention: 24–48h before a planned cutover, lower TTL to 60–300 seconds. After the change propagates, raise TTL back.

Q3. Your app sees many CLOSE_WAIT TCP connections. What does this indicate?

Show answer

CLOSE_WAIT means the remote side closed the connection (sent FIN) but your app hasn't called close() on the socket. Usually a bug — your application is holding connections open after the peer disconnects. Check for connection leaks, missing close() calls, or connection pool misconfigurations.

Q4. What's the difference between an AWS Security Group and a Network ACL? When would you use each?

Show answer

Security Groups: stateful, instance-level, allow-only rules, evaluated together. NACLs: stateless, subnet-level, numbered rules evaluated in order, support both allow AND deny. Use SGs for normal traffic control. Use NACLs as a subnet-level broad block (e.g., block an entire IP range across all instances in a subnet).
