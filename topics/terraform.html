Terraform

Level: Intermediate → Advanced  ·  Infrastructure as code for your AWS EKS clusters and cloud resources

How Terraform Works

Declarative model

You describe the desired end state. Terraform figures out what API calls to make to get there. "I want 3 EC2 instances" — Terraform creates/modifies/deletes to match.

State file

Terraform stores what it created in terraform.tfstate. Maps your config to real cloud resources. This is the source of truth — losing it means Terraform can't manage those resources anymore. Always store remotely (S3 + DynamoDB lock).

Providers

Plugins that talk to APIs — AWS, GCP, Azure, Kubernetes, GitHub, Datadog, etc. Each provider exposes resources and data sources. Versioned independently.

Plan & Apply

plan shows a diff — what will be created/modified/destroyed. apply executes it. Always review the plan before applying. In CI, save the plan and apply the exact saved plan.

Modules

Reusable units of infrastructure. A module is a directory of .tf files. Build internal modules for your VPC, EKS cluster, RDS setup — call them with different variables per environment.

DAG execution

Terraform builds a dependency graph from your references. Resources with no dependencies run in parallel. Resources that reference others (using resource.type.name.attribute) run after their dependencies.

HCL Syntax — Everything You Need

# ── PROVIDERS ────────────────────────────────────────────
terraform {
  required_version = ">= 1.7"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # allow 5.x, block 6.0
    }
  }

  # Remote state — ALWAYS use this in production
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/eks/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"   # prevents concurrent apply
  }
}

provider "aws" {
  region = var.region
  default_tags {
    tags = {
      Environment = var.environment
      ManagedBy   = "terraform"
      Team        = "platform"
    }
  }
}

# ── VARIABLES ─────────────────────────────────────────────
variable "environment" {
  type        = string
  description = "Deployment environment (dev/staging/prod)"
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "eks_node_count" {
  type    = number
  default = 3
}

# ── LOCALS ────────────────────────────────────────────────
locals {
  name_prefix = "${var.project}-${var.environment}"
  common_tags = {
    Project     = var.project
    Environment = var.environment
  }
}

# ── DATA SOURCES (read existing resources) ────────────────
data "aws_vpc" "existing" {
  tags = { Name = "${local.name_prefix}-vpc" }
}

data "aws_availability_zones" "available" {
  state = "available"
}

# ── RESOURCES ─────────────────────────────────────────────
resource "aws_eks_cluster" "main" {
  name     = "${local.name_prefix}-eks"
  role_arn = aws_iam_role.eks_cluster.arn
  version  = "1.29"

  vpc_config {
    subnet_ids              = aws_subnet.private[*].id
    endpoint_private_access = true
    endpoint_public_access  = false
  }

  depends_on = [
    aws_iam_role_policy_attachment.eks_cluster_policy,
  ]

  tags = local.common_tags
}

# ── OUTPUTS ───────────────────────────────────────────────
output "cluster_endpoint" {
  value     = aws_eks_cluster.main.endpoint
  sensitive = false
}

output "cluster_ca" {
  value     = aws_eks_cluster.main.certificate_authority[0].data
  sensitive = true
}

# ── COUNT / FOR_EACH ─────────────────────────────────────
# count: simple N copies
resource "aws_subnet" "private" {
  count             = length(data.aws_availability_zones.available.names)
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(var.vpc_cidr, 4, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

# for_each: map-based (preferred — stable addressing)
resource "aws_security_group_rule" "allow_services" {
  for_each = {
    "http"  = 80
    "https" = 443
    "grpc"  = 9000
  }
  type        = "ingress"
  from_port   = each.value
  to_port     = each.value
  protocol    = "tcp"
  description = "Allow ${each.key}"
  security_group_id = aws_security_group.app.id
  cidr_blocks = ["10.0.0.0/8"]
}

Calling a module:

module "eks" {
  source = "../../modules/eks"

  cluster_name    = "${local.name_prefix}-eks"
  cluster_version = "1.29"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnet_ids
  node_count      = var.environment == "prod" ? 5 : 2
  instance_types  = ["m5.xlarge"]

  tags = local.common_tags
}

Essential Commands

CommandWhat it does
terraform initDownload providers, initialise backend, set up modules
terraform validateSyntax check — fast, no API calls
terraform fmt -recursiveAuto-format all .tf files to canonical style
terraform plan -out=tfplanShow changes, save to file for apply
terraform apply tfplanApply the exact saved plan (CI/CD safe)
terraform destroy -target=resourceDestroy a specific resource only
terraform state listList all resources in state
terraform state show aws_eks_cluster.mainShow full attributes of a state resource
terraform import aws_s3_bucket.main my-bucketImport an existing resource into state without recreating
terraform taint aws_instance.webMark resource for recreation on next apply
terraform output -jsonGet all outputs as JSON — useful in CI pipelines
terraform graph | dot -Tsvg > graph.svgVisualise dependency graph
Never run terraform apply on production manually. Always: plan in CI → review plan → apply in CI with the saved plan file. Use Atlantis or Terraform Cloud for PR-based workflows.

# Create S3 bucket and DynamoDB table for state
aws s3api create-bucket --bucket my-tf-state-$(whoami) --region us-east-1
aws dynamodb create-table \
  --table-name terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \

  --billing-mode PAY_PER_REQUEST

Use the official EKS module (battle-tested)

# .github/workflows/terraform.yml
on:
  pull_request:
    paths: ['infra/**']

jobs:

    runs-on: ubuntu-latest

Check Your Understanding

Q1. Your team member ran terraform apply manually and now state is out of sync with what's in CI. How do you recover?

Show answer

1. Run terraform plan from CI — it will show the drift between state and actual infrastructure. 2. If the manual change is correct, run terraform import to bring the resource into managed state. 3. If you want to revert the manual change, run terraform apply from CI which will restore the desired state. Prevention: lock down AWS IAM so only the CI role can make changes; human users get read-only roles.

Q2. What is the difference between count and for_each? When should you use each?

Show answer

count creates N copies addressed by index (resource.name[0], resource.name[1]). Problem: if you remove index 1 from a list of 3, Terraform renumbers — resources[2] becomes [1] and gets destroyed/recreated. for_each addresses resources by key (resource.name["web"], resource.name["api"]). Removing one key only affects that specific resource. Prefer for_each whenever using a map or set — much safer for production changes.
